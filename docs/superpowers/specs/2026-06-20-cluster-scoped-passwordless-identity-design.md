# Cluster-scoped passwordless identity (full support) — design

Date: 2026-06-20
Status: Approved

## Context

PR #85 (`feat/azure-passwordless-47`) made a `TemporalCluster` reach `Ready`
against Azure Database for PostgreSQL Flexible Server with zero static passwords,
and added a real AKS + Flexible Server e2e (`test/e2e/azure`, `hack/azure-e2e.sh`)
that we have run green end-to-end. That first cut, however, has two limitations
this design removes:

1. **The operator authenticates its own probe/schema inspection in-process**
   (`internal/persistence/azure.go`, native Microsoft Entra token via
   `sql.azureWorkloadIdentity`). That token is acquired by the **operator**, with
   the **operator's** ServiceAccount in `temporal-system`, so it supports exactly
   **one** identity and couples database identity to the operator. Adding the
   operator's federated credential is a confusing, operator-scoped step
   (`AADSTS700213` if forgotten).
2. **A passwordless cluster requires extensive hand-written configuration** — the
   user must author `services.overrides.podTemplate` (sidecar, volume,
   container-by-name mount), `persistence.schemaJob.podTemplate` (token
   initContainer), and a `passwordCommandSecretRef` Secret. The example
   (`examples/cluster-azure-workload-identity`) is ~130 lines. This is not
   developer-friendly.

## Goals

- **Cluster-scoped, multi-identity:** the database identity is specified **by the
  cluster** and lives **in the cluster's own namespace** — not on the operator.
  Many clusters (and the future per-store case) can each use a different
  identity.
- **No operator-held database credential, no secrets storage:** the operator
  never acquires or stores a database token. All token acquisition happens in the
  cluster's namespace under the cluster's identity, via the generic
  `passwordCommand` mechanism — reusable by non-Azure providers.
- **Turnkey Azure abstraction:** a single high-level field expands into all the
  required wiring, so the example drops to a handful of lines.
- Keep the kind/nsc e2e flows intact; keep the Azure e2e green throughout.

## Non-goals (YAGNI)

- Per-store identities (cluster-level only for now; the design leaves room).
- Operator-side inspection for **arbitrary** `passwordCommand` providers beyond
  Azure (the inspector is generated by the Azure abstraction; the construction is
  shared so a future provider can reuse it).
- Continuous DB liveness probing from the operator (reachability is evaluated
  during active reconciliation and on spec changes; the Temporal pods surface
  ongoing outages). See "Trade-offs".
- Changing the static-password or Cassandra paths (unchanged).

## Design

### 1. API: one cluster-level field

Move Azure identity from per-store `SQLDatastoreSpec.azureWorkloadIdentity` to a
single cluster-level field on `PersistenceSpec`:

```yaml
spec:
  persistence:
    azureWorkloadIdentity:
      clientId: <managed-identity client id>   # required; drives the SA annotation
      scope: ...                               # optional; default ossrdbms .default scope
      serviceAccountName: ...                  # optional; default "<cluster>-azure"
      image: ...                               # optional; default mcr.microsoft.com/azure-cli:<pinned>
      refreshInterval: ...                     # optional; default 30m
```

- `tenantId` is intentionally absent: the Azure Workload Identity webhook injects
  `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, and `AZURE_FEDERATED_TOKEN_FILE` into any
  pod carrying the `azure.workload.identity/use: "true"` label, so the generated
  sidecars/initContainers read them from the environment.
- The existing generic escape hatch — `passwordCommandSecretRef`,
  `persistence.schemaJob.podTemplate`, `services.overrides` — **stays**. Azure
  users never touch it; other providers (GCP, Vault, …) wire their own sidecar +
  `passwordCommand` exactly as before, with no Azure coupling.

`sql.azureWorkloadIdentity` (per-store) is **removed**. Since the feature is
unreleased (introduced in this same PR), there is no migration concern.

### 2. Generated resources (the abstraction)

When `persistence.azureWorkloadIdentity` is set, the operator renders, **in the
cluster's namespace**:

- A **ServiceAccount** (default `<cluster>-azure`) with
  `azure.workload.identity/client-id: <clientId>`. Server pods, schema Jobs, and
  inspector Jobs all run as it and carry the `azure.workload.identity/use: "true"`
  pod label.
- **Server pods** (frontend/history/matching/worker): an `azure-token-refresher`
  sidecar (logs in via WI, writes `/azure/pgpass` on `refreshInterval`), a shared
  `emptyDir` token volume, the main container mounted at `/azure`, and an injected
  **`passwordCommand`** (`sh -c 'until [ -s /azure/pgpass ]; do sleep 1; done; cat
  /azure/pgpass'`). No user Secret — the command string is not sensitive and no
  token is stored.
- **Schema Jobs**: a one-shot `azure-token` initContainer writing `/azure/pgpass`,
  the volume + `schema` container mount, and the same injected `passwordCommand`.
- **Inspector Jobs**: same token initContainer + volume, plus a psql query
  container (see §3).

This wiring is built by a single internal helper so the server-pod, schema-Job,
and inspector-Job plumbing stay consistent. The same helper is the seam a future
non-Azure provider would reuse.

### 3. Operator probe/schema under model "D" (Job-based inspection)

`internal/persistence/azure.go` and the in-process token branch in
`internal/persistence/sql.go` (`resolvePassword`'s `AzureWorkloadIdentity` case)
are **deleted**. The operator becomes provider-agnostic. The `Backend` interface
(`Probe`, `SchemaVersion`, `EnsureSchema`) gains a second SQL implementation,
selected per store:

- **Static-password / existing paths — unchanged.** When the operator has a
  credential it can use directly (a static `passwordSecretRef`, or Cassandra), it
  connects **in-process** exactly as today (`DirectSQLBackend`). No behavior or
  performance change for the common case.
- **Command/Workload-Identity path — `JobInspectorBackend`.** When the credential
  is only obtainable via the cluster's identity (i.e. `azureWorkloadIdentity` is
  configured), the operator does **not** connect. Instead it runs a short-lived
  **inspector Job** per store in the cluster namespace. The Job's init container
  mints the token (azure-cli, cluster identity, writes `/azure/pgpass`); its main
  container is the **operator's own image** run with an `inspect` subcommand
  (`manager inspect --store <name> ...`). The inspector subcommand **reads the
  token from `/azure/pgpass`** (generic `passwordCommand`-style file read — no
  Azure SDK in the operator) and runs the operator's **existing**
  `SQLProber.Probe` + `CurrentSchemaVersion` Go code against:

  ```
  SELECT curr_version FROM schema_version WHERE db_name = '<db>'
  ```

  writing a small JSON result to the pod's **termination message**
  (`terminationMessagePath`), distinguishing three outcomes:
  - connect failure → `{"reachable": false, "error": "..."}`
  - connected, table absent → `{"reachable": true, "version": ""}` (needs setup)
  - version found → `{"reachable": true, "version": "1.13"}`

  Reusing the operator image means **no `psql`, no third image to mirror, and no
  duplicated query logic** (admin-tools ships no `psql` and `temporal-sql-tool`
  has no read-only version/ping command, so it cannot serve a read-only probe).
  The operator discovers its own image via the downward API (its pod's container
  image) so the inspector always matches the running operator. The operator reads
  `containerStatuses[].state.terminated.message`, parses it, and feeds the
  **existing** reconcile logic: `PersistenceReachable` from `reachable`,
  `Status.Persistence.SchemaVersions` from `version`, and the unchanged
  setup-vs-update decision from the version. The inspector Job uses
  `ttlSecondsAfterFinished` to self-clean; the backend runs it **once per
  reconcile** and caches `{reachable, version}` so `Probe` and `SchemaVersion`
  don't spawn two Jobs.

Because the inspector still reports the real schema version, the schema reconcile
(`buildSchemaTargets`, setup/update Jobs, `SchemaReady`) is **unchanged** — only
the *source* of `{reachable, version}` is swapped to the Job for the WI path.

**Inspection frequency (caching).** The operator runs the inspector when the
cluster is not yet `SchemaReady` or when `metadata.generation` changed since the
last successful inspection; otherwise it trusts the last result. This bounds Job
churn. (Continuous liveness probing is out of scope — see Trade-offs.)

### 4. Helm chart

Remove the operator-side `workloadIdentity.*` values and the label/SA-annotation
template (operator no longer needs a token). The operator pod becomes a plain
deployment again. Only the **cluster's** SA (in the cluster namespace) carries a
federated credential.

### 5. Example simplification (before → after)

`examples/cluster-azure-workload-identity/temporalcluster.yaml` collapses from
~130 lines of podTemplates/sidecars/secret to roughly:

```yaml
apiVersion: temporal.bmor10.com/v1alpha1
kind: TemporalCluster
metadata:
  name: azure-wi
spec:
  version: "1.31.1"
  numHistoryShards: 512
  persistence:
    azureWorkloadIdentity:
      clientId: REPLACE_WITH_MANAGED_IDENTITY_CLIENT_ID
    defaultStore:
      sql:
        pluginName: postgres12
        host: REPLACE_ME.postgres.database.azure.com
        port: 5432
        database: temporal
        user: temporal-identity
        tls: { enabled: true }
    visibilityStore:
      sql:
        pluginName: postgres12
        host: REPLACE_ME.postgres.database.azure.com
        port: 5432
        database: temporal_visibility
        user: temporal-identity
        tls: { enabled: true }
```

The separate `serviceaccount.yaml` is removed (the operator generates the SA).

### 6. E2E changes

- `test/e2e/azure/03-temporalcluster.yaml`: replace all podTemplates/sidecars with
  the single `persistence.azureWorkloadIdentity.clientId: ($values.clientId)`
  field; drop `01-serviceaccount.yaml` and `02-secret.yaml` (operator-generated).
  The assertion is unchanged (`PersistenceReachable=True`, `SchemaReady=True`).
- `hack/azure-e2e.sh`: the federated credential binds to the **cluster's**
  generated SA in the **cluster's** namespace
  (`system:serviceaccount:azure-e2e:azure-e2e-azure`, given a cluster named
  `azure-e2e`) instead of a hand-named SA; **remove the operator federated
  credential** and the operator `workloadIdentity.*` Helm flags. Everything else
  (region, Entra flags, db create, extension allow-list, firewall rule, PG16
  grants, cert-manager) stays.
- Keep validating by running `make azure-e2e-up && make azure-e2e-test &&
  make azure-e2e-down` (tearing down between iterations).

### 7. Testing

- **Unit:** the `inspect` subcommand emits the right `{reachable, version}` JSON
  for the three outcomes (connect failure / no-table / versioned) given a token
  file and DSN params (using the existing `SQLProber`/fake DB). `JobInspectorBackend`
  parses that JSON and drives the right conditions and setup-vs-update decision
  (fake Job client). The resource builder produces the expected SA, sidecar,
  initContainer, volume, mounts, injected `passwordCommand`, and inspector Job
  (operator image) from `azureWorkloadIdentity`. Static-password path unchanged
  (existing tests).
- **envtest/controller:** reconcile with `azureWorkloadIdentity` set creates the
  SA + inspector Job and surfaces conditions from a faked inspector result.
- **e2e:** the real Azure suite above.

## Trade-offs

- **Reachability is not continuously probed.** Under D the operator learns
  reachability from inspector Jobs during active reconciliation and on spec
  changes, not on a steady interval. A DB that fails *after* `SchemaReady` will
  surface through the Temporal pods, not a flipped `PersistenceReachable`. This is
  an accepted simplification (avoids per-interval Job churn); a future periodic
  re-inspection can be added if needed.
- **Inspector Job latency.** First reconcile of a WI cluster waits on a Job
  (seconds) rather than an in-process query. Acceptable for a one-time setup flow.
- **Per-store identity** is deferred; the cluster-level field can grow an optional
  per-store override later without breaking existing specs.

## What changes (summary)

- API: remove `SQLDatastoreSpec.azureWorkloadIdentity`; add
  `PersistenceSpec.azureWorkloadIdentity` (cluster-level). Regenerate CRDs +
  `dist/chart` CRDs by hand.
- Delete `internal/persistence/azure.go` and the in-process token branch.
- Add an `inspect` subcommand to the manager binary (`cmd/`) that reads the token
  file and runs the existing `SQLProber` probe + version query, emitting JSON to
  the termination message.
- Add `JobInspectorBackend` + the Azure wiring builder (SA, sidecar,
  initContainer, inspector Job using the operator image) in `internal/resources`.
- Controller: select backend per store; generate the SA + Azure pod wiring when
  `azureWorkloadIdentity` is set; discover the operator's own image via the
  downward API for the inspector Job.
- Helm chart: remove operator `workloadIdentity.*`.
- Example + Azure e2e: collapse to the single field; update `hack/azure-e2e.sh`
  federated-credential wiring.
