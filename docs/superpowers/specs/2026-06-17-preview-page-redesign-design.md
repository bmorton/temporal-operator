# Resource Preview Page Redesign — Design

**Date:** 2026-06-17
**Status:** Approved (design); ready for implementation planning
**Quality bar:** The preview tool may be **alpha**. Operator-side code must remain
high quality and behavior-preserving; the only Go change is the wasm shim's
return shape, fully covered by the existing build/test guard.

## Problem

The in-browser Resource Preview (GitHub Pages, alongside the Hugo docs) works:
it loads a standard-Go WebAssembly module compiled from the operator source, runs
client-side, accepts a pasted `TemporalCluster`, and renders every Kubernetes
object the operator would create. **The wasm and the generation logic work and
must be preserved.** The problem is purely presentation — the page is close to
unstyled HTML + vanilla JS.

This is a redesign of the **UI layer** plus one small change to **what the wasm
returns** (structured JSON instead of a phase-tagged blob). We do not rewrite the
operator or the generation logic, and we add no runtime server — output stays
drop-in static files emitted by Hugo to GitHub Pages.

## Current state (exploration findings)

- **Wasm entrypoint:** `cmd/preview-wasm/main.go` exports a global
  `temporalPreview(kind, yaml)` via `syscall/js` (`js.FuncOf` + `js.Global().Set`)
  returning a JSON string `{objects:[{kind,name,phase,yaml}], errors:[]}`. Backed
  by the pure `internal/plan` planner + `internal/webhook` defaulting/validation.
  Standard Go `GOOS=js GOARCH=wasm`; ~55 MB raw / **~9.7 MB gzipped**. Guarded by
  `cmd/preview-wasm/wasm_build_test.go` (`//go:build !js`, compiles the js/wasm
  target).
- **Page today:** a **standalone** static file `docs/static/preview/index.html`
  + `app.js` — its own `<html>`, *not* rendered through the hugo-book theme.
  Tabs-by-kind, `<details>` cards, hand-written CSS.
- **Build:** `hack/build-preview.sh` (`make preview-wasm`) compiles the wasm,
  copies `wasm_exec.js` from `$(go env GOROOT)/lib/wasm/wasm_exec.js`, and stages
  example `TemporalCluster` manifests + `examples/index.json` into
  `docs/static/preview/`. All generated files are git-ignored.
- **CI:** `.github/workflows/docs.yml` runs `make preview-wasm`, then Hugo
  (extended `0.140.2`) build + Pages deploy. Go pinned `1.26.x`; the repo targets
  Go `1.26.4`. The Hugo job does **not** currently install Tailwind/Node.
- **CR:** `TemporalCluster`, apiVersion `temporal.bmor10.com/v1alpha1`. Examples
  live under `examples/` (some multi-document).
- Theme `hugo-book` is a git submodule; site-level `docs/layouts` and
  `docs/assets` (which override/extend the theme) do not yet exist.

## Hard constraints (carried from the brief)

1. **Drop-in static.** Deploys as static assets on the existing Hugo/Pages flow.
   No runtime Node server. A build step is allowed only inside the Hugo/CI flow.
2. **Standard Go wasm.** Keep `GOOS=js GOARCH=wasm go build` with the standard Go
   compiler (not TinyGo — it can't compile `k8s.io/apimachinery` / `sigs.k8s.io`
   / controller-runtime). Multi-MB binary handled via lazy load + Pages gzip.
3. **Beautiful & responsive.** Polished; single-column / touch-friendly on mobile.
4. **Low maintenance.** Small surface area, few dependencies.

## Stack

- **Reactivity:** Alpine.js (single pinned CDN script, ~7 kB gz, no build).
- **CSS:** Tailwind v4 + daisyUI v5 plugin, compiled via **Hugo Pipes**
  (`css.TailwindCSS`), not the Tailwind CDN runtime. daisyUI supplies `card`,
  `collapse`, `badge`, `btn`, `tabs`, theming, and dark mode.
- **YAML highlighting:** highlight.js + `highlightjs-copy` plugin, paired
  `github` / `github-dark` themes. (Shiki only if quality proves inadequate.)
- **Wasm:** unchanged toolchain; lazy-loaded on first interaction;
  `wasm_exec.js` pinned to the CI Go version.

## Architecture & decisions

### 1. Page hosting — standalone Hugo-rendered layout

The playground needs full-bleed width, which the book theme's narrow content
column fights. The page becomes **Hugo-rendered** (so Hugo Pipes can process its
CSS/JS) but uses its **own standalone layout** that emits a complete `<html>`
document, bypassing the book chrome.

```
docs/
├── content/preview/_index.md      # front matter only; selects the layout
├── layouts/preview/list.html      # standalone <html>: input pane + output shell
├── assets/
│   ├── css/preview.css            # Tailwind v4 entry + @plugin daisyUI (Hugo Pipes)
│   └── js/preview.js              # Alpine init, lazy wasm loader, card rendering
└── static/preview/                # generated by build-preview.sh (git-ignored)
    ├── temporal-operator-preview.wasm
    ├── wasm_exec.js
    └── examples/ (+ index.json)
```

`content/preview/_index.md` uses `type = "preview"` + `layout = "list"` so Hugo
renders `layouts/preview/list.html`. The layout pulls hashed CSS/JS through Hugo
Pipes:

```go-html-template
{{ $css := resources.Get "css/preview.css" | css.TailwindCSS | fingerprint }}
{{ $js  := resources.Get "js/preview.js"  | fingerprint }}
```

The old `docs/static/preview/index.html` + `app.js` are removed (the build script
no longer needs to coexist with a hand-maintained HTML file). `resource-preview.md`
and `tools/_index.md` continue to link to `/preview/` and are updated to drop the
"tabs" wording in favour of the new grouped-card description.

### 2. Tailwind via Hugo Pipes + CI

`assets/css/preview.css` is a Tailwind v4 entry:

```css
@import "tailwindcss";
@plugin "daisyui";
```

`css.TailwindCSS` invokes the Tailwind v4 CLI at Hugo build time, which purges to
only the classes used by `layouts/` + `assets/js/`, producing a small fingerprinted
stylesheet. This requires the Tailwind CLI + daisyUI to be installed before Hugo
runs.

- A minimal `docs/package.json` pins `tailwindcss`, `@tailwindcss/cli`, and
  `daisyui`. A `docs/postcss`-free setup — Tailwind v4 needs no PostCSS config.
- The docs workflow's `build-deploy` job gains a `setup-node` + `npm ci`
  (working-directory `docs`) step **before** the Hugo build. The `preview-wasm`
  PR-check job is unaffected.
- Tailwind binary is discovered by Hugo via `node_modules/.bin`; no global install.

Rationale: this is the "light build step that runs inside the existing Hugo/CI
flow" the brief permits, and keeps the shipped CSS purged and small (no CDN
runtime).

### 3. Wasm JSON contract

Keep the export name `temporalPreview(kind, yaml)` (only `TemporalCluster` is
wired today) and keep `syscall/js` (`js.FuncOf` + `js.Global().Set`; **no**
`//go:wasmexport`). Only the **return shape** changes, to the brief's structured
contract (extended with the already-useful `phase`):

```go
type previewResource struct {
    Kind       string `json:"kind"`
    APIVersion string `json:"apiVersion"`
    Name       string `json:"name"`
    Namespace  string `json:"namespace"` // "" for cluster-scoped
    Phase      string `json:"phase"`     // operator reconcile phase (retained)
    YAML       string `json:"yaml"`      // rendered manifest for this one object
}

type previewResult struct {
    Resources []previewResource `json:"resources"`
    Error     string            `json:"error,omitempty"`
}
```

- `apiVersion` and `namespace` are read from each planned object's
  `GroupVersionKind` / `GetNamespace()`. Cluster-scoped objects emit `""`.
- Generation/parse failures populate `Error` (a single string; multiple internal
  messages are joined) instead of panicking. On error, `Resources` is `[]`.
- `wasm_build_test.go` stays a pure compile guard; no contract assertions needed
  there. (Existing behavior preserved: secret `data` is rendered as readable
  `stringData`; multi-doc input still scans for the first `TemporalCluster`.)

### 4. Cache-busting the fixed-name wasm

Hugo only fingerprints `assets/` via Pipes, and a ~55 MB binary should not go
through Pipes. The `.wasm` stays in `static/preview/` (Hugo copies it verbatim,
fast). `build-preview.sh` writes the binary's content hash to
`docs/data/preview.json` (`{"wasmHash":"<sha256-prefix>"}`); the layout reads
`site.Data.preview.wasmHash` and requests `temporal-operator-preview.wasm?v=<hash>`
so a CI rebuild busts the browser cache. `docs/data/preview.json` is git-ignored
like the other generated preview assets.

## UI (Stage 1)

Single-column on mobile, two-pane on desktop. All reactivity via Alpine; one
`x-data` component (`previewApp()` in `assets/js/preview.js`).

**Input pane**
- Labeled `<textarea>` for the CR YAML (textarea in Stage 1; CodeMirror is Stage 2).
- "Load example…" `<select>` populated from `static/preview/examples/index.json`;
  selecting injects that manifest's text.
- "Preview" `btn`, disabled until wasm is ready; shows a spinner while wasm loads
  and while generating.
- An error banner (daisyUI `alert`) rendering `previewResult.error` when present.

**Output pane (the feature that matters)**
- Resources **grouped by `kind`**, each group with a section header + count
  ("Deployments (2)", "Services (3)", …), kinds sorted alphabetically.
- **One collapsible card per resource** (daisyUI `collapse` over native
  `<details>`), collapsed by default.
- Card header: **color-coded kind badge**, `metadata.name`, and `namespace` when
  present. Badge color is derived from a small kind→color map with a stable
  fallback hash so new kinds still get a consistent color.
- Card body: highlighted YAML for that single object + a **copy-to-clipboard**
  button (`highlightjs-copy`).
- Global controls above output: **Expand all / Collapse all**, a **kind filter**
  (toggle which kinds show), and **Copy all / Download all** (concatenate visible
  resources into a `---`-separated multi-doc YAML; download as `preview.yaml`).
- **Lazy highlight:** highlighting runs for a card only when it is first expanded
  (cache the result), keeping large previews fast.

**Polish / behavior**
- Dark-mode toggle via daisyUI theme (`light` ⇄ `dark`); swap highlight.js
  `github` ⇄ `github-dark` stylesheet to match.
- Mobile: panes stack to one column; cards full-width; code blocks scroll
  horizontally rather than overflow.
- Loading UX: a skeleton / determinate-ish indicator for the multi-MB wasm fetch
  (use `Content-Length` when present), not a bare spinner.

**Wasm loading**
- **Lazy:** do not fetch `.wasm` on page load. Fetch on first interaction (first
  "Preview" click or first focus into the textarea), with a loading indicator.
- Load `wasm_exec.js` before app JS; instantiate via
  `WebAssembly.instantiateStreaming(fetch(url), go.importObject)` with a
  non-streaming `arrayBuffer` fallback (older Pages MIME edge cases).
- Wrap `temporalPreview(...)` calls in try/catch; render failures in the banner.
- GitHub Pages serves `application/wasm` + gzips automatically; no extra config.
  (Brotli is not available on Pages.)

## Out of scope (later stages)

- **Stage 2:** replace textarea with lazy-loaded CodeMirror 6 (YAML) + optional
  client-side YAML validation.
- **Stage 3:** shareable links (lz-string into the URL fragment) and a diff view.
- Web Worker offload (only if generation jank appears; not needed for Stage 1).

## Acceptance criteria (Stage 1)

Pasting or loading an example `TemporalCluster` renders all generated objects as
**kind-grouped, collapsible, individually copyable cards** with syntax-highlighted
YAML; **expand/collapse-all**, **kind filter**, and **copy/download-all** work;
the page is fast and clean on mobile; the wasm **loads lazily** with a clear
loading state; errors surface in the banner. The js/wasm build still passes
(`make preview-wasm`, `go test ./cmd/preview-wasm/...`), and the docs site builds
in CI with the new Tailwind step.

## Risks

- **CI Tailwind step:** adds Node + `npm ci` to the docs deploy. Mitigation: pin
  versions in `docs/package.json` + lockfile; scope the npm step to `docs/`.
- **Wasm size:** ~9.7 MB gz. Mitigation: lazy load + determinate loading UI +
  cache-busted filename so repeat visits hit cache.
- **daisyUI/Tailwind v4 churn:** pin exact versions; the surface used is small.
