# Design: Host documentation on GitHub Pages with Hugo

## Status

Approved (2026-06-14).

## Problem

The repository already contains documentation content under `docs/content/`,
laid out in Hugo's section convention (`_index.md` files, relative section
links such as `./getting-started/`). However:

- There is no static-site-generator configuration, so nothing builds a site.
- The `docs.yml` workflow only lints Markdown and checks links. Its trailing
  comment notes that production deployment "requires a configured Hugo/Docsy
  site; wire it here when available."
- The CRD reference (`docs/api/v1alpha1.md` and `docs/content/reference/`) is
  generated by `crd-ref-docs`.

We want a published documentation site hosted on **GitHub Pages**.

## Constraints & context

- **Repository is private, owner is on a paid GitHub plan.** GitHub Pages is
  available for private repositories on paid plans. Note: on GitHub Pro,
  Pages *access control* (restricting viewers to people with repo access) is
  not available, so the published site will be publicly reachable at its URL
  even though the repository stays private. This is acceptable for docs.
- Project (not user) Pages site, served at
  `https://bmorton.github.io/temporal-operator/`.
- Generated files (`docs/api/**`, `docs/content/reference/**`) are excluded
  from markdownlint/lychee today and must not be hand-edited.

## Chosen approach

Hugo with the lightweight **`hugo-book`** theme, deployed to GitHub Pages via
GitHub Actions. Rationale: the existing content is already Hugo-shaped, so this
is the least-rework option; the build is a single Hugo binary (fast CI); and we
can graduate to a heavier theme (e.g. Docsy) later if needed.

Alternatives considered and rejected:

- **Hugo + Docsy** — richer (versioning, search), but heavyweight build (Hugo
  Extended + Node/PostCSS + theme as a Hugo module). More moving parts than we
  need now.
- **MkDocs Material** — polished and popular, but the content is laid out for
  Hugo; adopting it means restructuring files (no `_index.md`, different link
  conventions, nav defined in `mkdocs.yml`). Most rework.

`hugo-book` requires **Hugo Extended** (it compiles SCSS).

## Components

### 1. Site configuration

- `docs/hugo.toml` — site root is `docs/` (content already in `docs/content/`):
  - `baseURL = "https://bmorton.github.io/temporal-operator/"`
  - `theme = "hugo-book"`
  - `title = "temporal-operator"`
  - Book params: table of contents, repository/edit links pointing at
    `https://github.com/bmorton/temporal-operator`.

### 2. Theme delivery

- `hugo-book` added as a git submodule at `docs/themes/hugo-book`, pinned to a
  released tag. Adds a `.gitmodules` entry. CI checks out with
  `submodules: recursive`.

### 3. Content preparation (small, mechanical)

- Add minimal TOML front matter (`title`, `weight`) to the section `_index.md`
  files and the home page so the left-sidebar nav has stable titles and order.
  Body content is untouched. Affected files (~10):
  - `docs/content/_index.md`
  - `docs/content/getting-started/_index.md`
  - `docs/content/installation/_index.md`
  - `docs/content/architecture/_index.md`
  - `docs/content/operations/_index.md`
  - `docs/content/upgrades/_index.md`
  - `docs/content/troubleshooting/_index.md`
  - `docs/content/contributing/_index.md`
  - `docs/content/install/verifying-releases.md` (leaf page)
- Add `docs/content/reference/_index.md` (a section landing page with front
  matter) so the CRD reference has a titled nav entry.
- Ensure the `crd-ref-docs`-generated `crds.md` renders with a title **without
  hand-editing generated output** — accomplished via the generator
  configuration (`hack/crd-ref-docs-config.yaml`) or a section `_index.md` that
  provides the heading, whichever is cleaner. Generated files remain excluded
  from lint.
- `docs/api/v1alpha1.md` stays outside `content/`; Hugo ignores it. No change.

### 4. CI/CD — extend `.github/workflows/docs.yml`

- Keep the existing **lint** job (runs on PRs and pushes).
- Add a **build-deploy** job, gated to **push on `main`**:
  1. `actions/checkout` with `submodules: recursive`.
  2. Setup Hugo Extended (pinned action + version).
  3. `hugo --minify` (site root `docs/`, output `docs/public`).
  4. `actions/configure-pages`.
  5. `actions/upload-pages-artifact` with `path: docs/public`.
  6. `actions/deploy-pages`.
- Job-level permissions: `pages: write`, `id-token: write`, `contents: read`.
- `concurrency: { group: pages, cancel-in-progress: false }`.
- `environment: github-pages`.
- Remove the stale Cloudflare/"wire it here when available" comment.

### 5. One-time Pages enablement

- Set the repository Pages build source to **GitHub Actions** via the API
  (`gh api`), so it is configured rather than only documented.

### 6. Local development & ignores

- Makefile targets:
  - `docs-serve` → `hugo server -s docs`
  - `docs-build` → `hugo -s docs --minify`
  - Document the Hugo Extended prerequisite.
- `.gitignore` additions: `docs/public/`, `docs/resources/`, `.hugo_build.lock`.

## Out of scope (YAGNI for now)

- Custom domain.
- Site search.
- Documentation versioning.
- Dark-mode / theme customization beyond defaults.

These are straightforward to add later and are intentionally deferred.

## Success criteria

- `make docs-build` (or `hugo -s docs`) builds the site locally with no errors
  using Hugo Extended.
- On push to `main`, the `docs.yml` build-deploy job publishes the site and it
  is reachable at `https://bmorton.github.io/temporal-operator/`.
- The left-sidebar navigation lists the documentation sections in a sensible
  order with readable titles.
- Existing Markdown lint/link-check behavior is preserved; generated files are
  still excluded and not hand-edited.
